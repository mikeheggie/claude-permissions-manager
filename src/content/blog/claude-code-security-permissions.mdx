## Why Security Matters with AI Coding Assistants

AI coding assistants like Claude Code are incredibly powerful, but that power comes with responsibility. Claude can read files, execute commands, and make changes across your codebase. Without proper guardrails, it could accidentally expose secrets, delete important files, or execute commands you didn't intend.

The good news: Claude Code has a robust permission system that lets you define exactly what it can and cannot do. This guide shows you how to configure security-first permissions.

## The Deny-First Approach

Claude Code evaluates permissions in this order:

1. **Deny rules checked first** - If a deny rule matches, the action is blocked
2. **Ask rules checked second** - If an ask rule matches, Claude prompts for confirmation
3. **Allow rules checked last** - If an allow rule matches, the action proceeds

This means deny rules always win. You can have broad allow patterns while using deny rules to block specific dangerous variations.

```json
{
  "permissions": {
    "allow": [
      "Bash(git:*)"
    ],
    "deny": [
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)"
    ]
  }
}
```

In this example, most git commands are allowed, but force push and hard reset are blocked.

## Protecting Sensitive Files

### Environment Files

Your `.env` files contain API keys, database credentials, and other secrets. Block access to these files:

```json
{
  "permissions": {
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Read(**/.env)",
      "Read(**/.env.*)",
      "Edit(.env)",
      "Edit(.env.*)",
      "Edit(**/.env)",
      "Edit(**/.env.*)"
    ]
  }
}
```

### Credentials and Secrets

Extend protection to other sensitive files:

```json
{
  "permissions": {
    "deny": [
      "Read(**/credentials*)",
      "Read(**/secrets*)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Read(**/*_rsa)",
      "Read(**/*_rsa.pub)",
      "Read(**/.ssh/*)",
      "Read(**/id_*)"
    ]
  }
}
```

### Configuration Files with Secrets

Some config files may contain embedded secrets:

```json
{
  "permissions": {
    "deny": [
      "Read(**/config.production.*)",
      "Read(**/secrets.json)",
      "Read(**/secrets.yaml)",
      "Read(**/vault.json)"
    ]
  }
}
```

## Blocking Dangerous Commands

### Destructive Operations

Block commands that can cause irreversible damage:

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(rm -r /*:*)",
      "Bash(rmdir:*)",
      "Bash(sudo rm:*)",
      "Bash(> /dev/:*)",
      "Bash(dd if=:*)"
    ]
  }
}
```

### System-Level Commands

Block commands that modify system state:

```json
{
  "permissions": {
    "deny": [
      "Bash(sudo:*)",
      "Bash(su :*)",
      "Bash(chmod 777:*)",
      "Bash(chown:*)",
      "Bash(systemctl:*)",
      "Bash(service :*)"
    ]
  }
}
```

### Network and External Access

Control what can access the network:

```json
{
  "permissions": {
    "deny": [
      "Bash(curl:*)",
      "Bash(wget:*)",
      "Bash(ssh :*)",
      "Bash(scp :*)",
      "Bash(rsync:*)"
    ]
  }
}
```

If you need curl for specific purposes, use allow rules for specific URLs while keeping the broad deny:

```json
{
  "permissions": {
    "allow": [
      "Bash(curl https://api.github.com:*)",
      "Bash(curl https://registry.npmjs.org:*)"
    ],
    "deny": [
      "Bash(curl:*)"
    ]
  }
}
```

### Publishing and Deployment

Prevent accidental publishes:

```json
{
  "permissions": {
    "deny": [
      "Bash(npm publish:*)",
      "Bash(yarn publish:*)",
      "Bash(pnpm publish:*)",
      "Bash(cargo publish:*)",
      "Bash(gem push:*)",
      "Bash(pip upload:*)",
      "Bash(twine upload:*)"
    ]
  }
}
```
